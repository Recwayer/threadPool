# Кастомный пул потоков для высоконагруженных серверных приложений

## Описание проекта

Проект реализует собственный **CustomThreadPool**, который позволяет:

- Гибко управлять количеством потоков (`corePoolSize`, `maxPoolSize`).
- Настраивать очередь задач (`queueSize`) и время простоя потоков (`keepAliveTime`).
- Поддерживать **резервные потоки** (`minSpareThreads`), готовые к обработке задач.
- Использовать различные **алгоритмы балансировки нагрузки**:
    - Round Robin
    - Least Loaded
- Реализовать **политику отказа** при переполнении очередей:
    - AbortPolicy
    - CallerRunsPolicy
- Вести подробное логирование всех ключевых событий.
- Использовать интерфейс `CustomExecutor`, что позволяет работать с `Runnable` и `Callable`.

---

## Основные компоненты

- **CustomThreadPool** - главный управляющий компонент.
- **Worker** - поток, выполняющий задачи из закрепленной очереди.
- **TaskQueue** - очередь задач на основе `ArrayBlockingQueue`.
- **CustomThreadFactory** - фабрика потоков с логированием и уникальными именами.
- **ExecutionBalancer** - алгоритм распределения задач между очередями.
- **RejectionPolicy** - политика обработки задач при перегрузке.

---
## Принцип распределения задач и балансировки

Каждому worker’у соответствует **собственная** `TaskQueue`. При поступлении задачи `CustomThreadPool` через `ExecutionBalancer` выбирает подходящую очередь:

- **RoundRobinBalancer** (по умолчанию) - циклически распределяет задачи между очередями.  
  **Почему выбран:** Простой, предсказуемый и справедливый алгоритм при однородных задачах.  
  **Недостатки:** Может привести к неравномерной нагрузке, если задачи имеют сильно различающуюся длительность.

- **LeastLoadedBalancer** - выбирает очередь с **наименьшим** текущим размером (`size()`).  
  **Преимущество:** Лучше балансирует нагрузку при разнородной длительности задач и предотвращает перегрузку отдельных рабочих потоков.  
  **Недостаток:** Требует перебора всех очередей при каждом поступлении задачи.

Алгоритм выбирается через метод `setExecutionBalancer()`.

---

## Политика отказа в обработке задач

При переполнении всех очередей и достижении `maxPoolSize` вызывается `RejectionPolicy`:

- **AbortPolicy** (по умолчанию) - выбрасывает `RejectedExecutionException`.  
  **Почему выбран:** гарантирует, что ни одна задача не потеряется молча и приложение получит явный сигнал о перегрузке.  
  **Недостатки:** требует обработки исключения на стороне вызывающего кода.

- **CallerRunsPolicy** - задача выполняется в потоке вызывающего метода.  
  **Преимущество:** Система не падает и выполняет задачу в producer-потоке.
  **Недостаток:** может замедлить producer-поток и привести к каскадному замедлению.

Политика выбирается через метод `setRejectionPolicy()`.

## Анализ производительности

**Сравнение с `java.util.concurrent.ThreadPoolExecutor`:**

| Параметр                    | CustomThreadPool                     | ThreadPoolExecutor               | Комментарий                          |
|-----------------------------|--------------------------------------|----------------------------------|--------------------------------------|
| Очереди                     | По одной на worker                   | Одна общая очередь               | Лучшая изоляция и предсказуемость   |
| Балансировка                | RoundRobin / LeastLoaded             | Отсутствует (FIFO)               | Значительно выше гибкость            |
| `minSpareThreads`           | Есть                                 | Отсутствует                      | Уникальная фича                      |
| Overhead на `submit`        | Чуть выше (балансировка)             | Минимальный                      | Приемлемо для большинства задач      |
| Логирование                 | Подробное (все ключевые события)     | Минимальное                      | Удобно для отладки и мониторинга    |

`CustomThreadPool` предоставляет **большую гибкость** за счёт отдельных очередей, сменяемых балансировщиков и политики `minSpareThreads`.  
В отличие от стандартного `ThreadPoolExecutor` он позволяет динамически менять стратегию распределения задач и политику отказа.  
По архитектуре ближе всего к пулам потоков в Tomcat и Jetty, где также используются отдельные очереди и настраиваемые политики обработки перегрузки.

## Мини-исследование оптимальных параметров

На основе теории и экспериментов с `Main.java`:

| Параметр          | Рекомендуемое значение   | Почему                                                                                                                        |
|-------------------|--------------------------|-------------------------------------------------------------------------------------------------------------------------------|
| `corePoolSize`    | ≈ количество ядер CPU    | Основная нагрузка                                                                                                             |
| `maxPoolSize`     | `corePoolSize * 2…4`     | Избежать overhead контекстного переключения                                                                                   |
| `queueSize`       | 50-500                   | Маленькая очередь позволяет быстрее реагировать на перегрузку.<br>Большая очередь может привести к длительному ожиданию задач. |
| `keepAliveTime`   | 30-60 секунд             | Оптимальное время жизни неиспользуемых потоков в большинстве сценариев                                                        |
| `minSpareThreads` | 1-2 сверх `corePoolSize` | Держит "горячие" потоки под небольшой нагрузкой                                                                                |

**Лучшие показатели производительности** достигаются при:
- небольшой очереди + `AbortPolicy`,
- `LeastLoadedBalancer` при разнородных задачах,
- `minSpareThreads >= corePoolSize` (снижает количество созданий и удалений потоков).
